### 1. 漏斗与阀门：`asyncio.Semaphore` (信号量)

- **工程直觉**：想象一个只有 5 个连接通道的 LLM 供应商 API。你有 1000 个任务要发，你必须在网关层加一个“限流阀”。`Semaphore` 就是一个拥有固定数量“通行证”的保安。
- **核心语法**：

```python
# 初始化一个只有 5 张通行证的信号量
sem = asyncio.Semaphore(5)

async def limited_task():
    # 必须拿到通行证才能往下走，拿不到当前协程就会被挂起等待
    async with sem:
        await call_fragile_upstream_api()
    # 离开 with 块时，通行证会自动归还

```

### 2. 传送带与打工人：`asyncio.Queue` (异步队列)

- **工程直觉**：当你的系统分为“接收任务（极快）”和“处理任务（极慢）”两部分时，绝对不能让它们直接对接。你需要一条有缓冲能力的传送带。这在解耦“大模型请求网关”和“后台推理节点”时极其常用。
- **核心语法**：

```python
    queue = asyncio.Queue(maxsize=100) # 限制缓冲池大小，防止内存暴涨

    # 生产者
    await queue.put(item) # 如果传送带满了 100 个，生产者会自动挂起等待

    # 消费者
    item = await queue.get() # 如果传送带空了，消费者会自动挂起等待
    queue.task_done() # 关键：处理完一个，必须告诉队列一声
```

### 3. 批处理与流水线：`gather` vs `as_completed`

- **工程直觉**：
  - `asyncio.gather(*tasks)` 就像期末考试，必须等所有人（所有协程）都交卷，才能拿到一张按座位号排好序的成绩单。
  - `asyncio.as_completed(tasks)` 则是流水线模式，谁先交卷，谁的成绩单就先弹出来。在流式输出（Streaming）或者需要尽早将结果写库的场景中，这是提效神器。
- **核心语法**：

```python
    # as_completed 返回的是一个迭代器，按完成的先后顺序产出结果
    for coro in asyncio.as_completed(tasks):
        result = await coro
        print(f"一个任务完成了，赶紧先处理：{result}")
```
